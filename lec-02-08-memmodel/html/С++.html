Memory location

bit-fields:

struct T {

Access memory location

Два потока пытаются работать с одним и тем же участком памяти (memory location).

По крайней мере один поток пытается изменить память.

Чтобы исключить data race, достаточно выполнения одного из условий:

  • операция модификации должна быть атомарна

  • одна операция должна быть выполнена до другой (happens-before)

Memory model

  • атомарность операций

  • частичное упорядочивание операций

  • видимость эффектов операций в других потоках

Atomics

По умолчанию sequential consistency:

  • Порядок выполнения операций совпадет с порядком в коде

  • Существует глобальный порядок всех операций всех потоков

ВАЖНО: правило относится к атомарным операциям

atomic<int> x;
int a, b, c;

int main() {
    // ...
    x.store(10);    // всегда до `c = x.load()`

    a = 10;
    b = 23;

    c = x.load();   // всегда после `x.store(10)`
    // ...
}

Atomics (sequential consistency)

#include <atomic>
#include <thread>

std::atomic<int> x, y;
int resY, resX;

void jobA() {
    x.store(42);
    resY = y.load();
}

void jobB() {
    y.store(42);
    resX = x.load();
}

int main() {
    std::thread thA{jobA}, thB{jobB};
    thA.join(); thB.join();
    return 0;
}

Atomics (sequential consistency)

// ...
void jobA() {
    x.store(42);           // (axs)
    resY = y.load();       // (ayl)
}

void jobB() {
    y.store(42);           // (bys)
    resX = x.load();       // (bxl)
}
// ...
Возможна только одна из последовательностей операций:
(axs) -> (ayl) -> (bys) -> (bxl)
(axs) -> (bys) -> (ayl) -> (bxl)
(axs) -> (bys) -> (bxl) -> (ayl)
(bys) -> (bxl) -> (axs) -> (ayl)
(bys) -> (axs) -> (ayl) -> (bxl)
(bys) -> (axs) -> (bxl) -> (ayl)

Глобальность: все потоки видят операции в одном и том же порядке.

    * может увеличиться количество вариантов последовательностей

Sync and Ordering

typedef enum memory_order {
    memory_order_relaxed,
    memory_order_consume,
    memory_order_acquire,
    memory_order_release,
    memory_order_acq_rel,           // acquire release semantic
    memory_order_seq_cst            // Default: sequential consistency
} memory_order;

Atomic Ops Taxonomy & Memory Order

                op              memory_order

READ        | load          |  acquire, consume

WRITE       | store         |  release

READ-WRITE  | test_and_set  |  acq_rel, seq_cst
            | exchange      |
            | fetch_*       |

Эквивалентность операций

load(memory_order_acq_rel) ~ load(memory_order_acquire)
load(memory_order_release) ~ load(memory_order_relaxed)

Sync & Ordering

  • sequential consistency: глобальный порядок

  • acquire-release: порядок операций в рамках одного атомика

  • relaxed: только атомарность и порядок модификации

Acquire Release Semantic

Синхронизация операций для одного и того атомика в разных потоках

  • Операция записи в атомик в потоке1 синхронизируется с операцией чтения в потоке2.

Acquire Release Semantic (ARS)

Идея:

  • Все операции чтения\записи не могут быть "перемещены" после release-операции

  • Все операции чтения\записи не могут быть "перемещены" до acquire-операции

ARS & Mutexes

std::mutex mtx
int a;

void job() {
    mtx.lock(); // acquire
    a += 1;
    mtx.unlock(); // release
}
*) cv: wait - acquire, notify_* - release
*) thread: join - acquire, completion - release

Acquire Release Semantic

    let thA: m.store(42, memory_order_release)

    let thB: m.load(memory_order_acquire)

    Пусть thB в load считывает "тот самый" 42

    Тогда можно считать этот load ТОЧКОЙ СИНХРОНИЗАЦИИ:

        Все операции с записью памяти в thA, которые произошли до store,
        становятся видимыми в thB

Spinlock example

class Spinlock {
    std::atomic_flag locked_;
public:
    Spinlock() : locked_(ATOMIC_FLAG_INIT) {}
    void lock() {
        while (locked_.test_and_set(memory_order_acquire));
    }
    void unlock() {
        locked_.clear(memory_order_release);
    }
};

Spinlock m;
void job() {
    m.lock();  // acquire
    // critical section
    m.unlock();  // release
}
*) Если реализовать через seq_cst, то будет меньше свободы для оптимизаций!

Transitive Sync

int data;
atomic<bool> dataReadyToTake1(false);
atomic<bool> dataReadyToTake2(false);

void producer() {
    data = 100;
    dataReadyToTake1.store(true, std::memory_order_release);
}

void carrier() {
    while (!dataReadyToTake1.load(std::memory_order_acquire));
    dataReadyToTake2.store(true, std::memory_order_release);
}

void consumer() {
    while (!dataReadyToTake2.load(std::memory_order_acquire));
    process(data);
}

Release-Consume Simplified

atomic<int> p;
int d;

void producer() {
    d = 100;
    p.store(42, std::memory_order_release);
}

void consumer() {
    int pp;
    while (!(pp = p.load(std::memory_order_consume)));
    std::cout << pp << std::endl; // OK, data dependency works
    std::cout << d << std::endl; // Data Race [UB]
}
*) Can be not supported with fallback to acquire

Relaxed Semantic

  • атомарность

  • существует некоторый глобальный порядок модификаций

  • нет синхронизаций

  • нет ограничений на порядок

    Usage: счетчики (например инкеремент shared_ptr)

Relaxed Semantic Example

atomic<int> counter;
void add() {        // wait-free
    for (int i = 0; i < 10000; ++i) {
        counter.fetch_add(1, std::memory_order_relaxed);
    }
}

int main() {
    vector<thread> ths;
    for (int i = 0; i < 100; ++i) {
        ths.emplace_back(add);
    }
    for (auto& th : ths) {
        th.join();
    }
    std::cout << counter << std::endl;
}
