Relocation

  • процесс подстановки нужных адресов вместо символов

  • секции с использованиями: .rela.data, .rela.text

  • секции с определениями (и адресами): .symtab

    • dynamic linking information

    • можно посмотреть через objdump

linking statically

  • релокация происходит во время линковки

    • результат — единый файл

  • можно линковаться с уже готовой статической библиотекой

  • файлы: libxxx.a в Linux, xxx.lib в Windows

linking dynamically

  • релокация отложена (на время исполнения)

    • результат — в exec появляется информация о зависимости

  • файлы: libxxx.so в Linux, xxx.dll в Windows

    • в них — необходимые определения + информация о динамической линковке

  • символы в адрес надо будет маппить в рантайме

    • на старте программы/при загрузке библиотеки/при вызове

link-time optimizations [*]

  • ограничения оптимизаций при компиляции — внешние интефейсы (boundaries)

  • идея: оптимизировать и их, на этапе линковки

    • объектные файлы .o: машинный код или Intermediate Language (IL)

    • + применение бекенда-оптимизатора

  • MSVC: Link Time Code Generation, details

  • LLVM: Link Time Optimization (libLTO), doc with example

static & dynamic libraries

static library

  • архив с объектными файлами

  • линкуется с программой до запуска

  • при запуске разных программ одинаковые либы не переиспользуются

    • при одновременном исполнении — дублирование данных

dynamic library

  • формат — похож на исполняемый файл

  • подтягивается во время исполнения

    • есть интерфейс для отложенной/ленивой загрузки (плагины)

  • в рантайме по возможности переиспользуется между процессами

    • код из библиотеки — переиспользуется, данные — свои (странички создаются CoW)

static vs dynamic

  • s > d: код влинковывается ⇒ нет проблем с зависимостями в рантайме

  • s > d: попадает только необходимый библиотечный код (с оптимизациями[*])

  • d > s: размер файла/process image

  • d > s: переиспользование кода

  • d > s: Interposition

Interposition

Interposition — возможность прозрачно заменить зависимость в среде исполнения

Процесс работы с внешними символами различный:

Windows: required symbols

Секция .idata — по-библиотечная информация про символы

  • символ берется из конкретной библиотеки

Linux: required symbols

Секции .dynamic + .dynsym — отдельно набор библиотек, отдельно набор символов

  • дизайн линковки: можно искать символ во всех SO (+ порядок обхода)

  • возможен Interposition: иметь разные реализации void f()

    • хотя это и нарушает ODR

    • с оптимизациями на стыке — сложно

Linux: required symbols

  • overhead: разный вызов на f(); (в зависимости от набора библиотек) ⇒

    • часть вызовов функций использует indirect (непрямой вызов через доп адрес)

    • "Аналог vtable": Global Offset Table

  • картинка на доске (?)

Linux: lazy symbols loading (Delayed)

  • Идея: разрешать не все символы сразу, а по мере использования

    • Обращение за символом — в доп таблицу Procedure Linkage Table

    • Адреса внутри GOT перезаписываемы (SecTrigger)

    • Исхдные адреса — на код с семантикой "Loader, найди символ и перезапиши адрес"

  • картинка на доске (?)

Position Independent Code [*]

  • Reason: SO reusage + security issues

  • Linux: -fPIC

  • For executable: -fPIE

Libraries usage

creating: static library

  • MSVS: изменить тип таргета (Static Library, .lib)

  • Linux:

$ g++ -Wall -c *.cpp
$ ar r libsome.a *.o

creating: dynamic library

  • MSVS: изменить тип таргета (Dynamic Library, .dll)

  • Linux:

$ g++ -Wall -c *.cpp
$ g++ -shared -o libsome.so *.o

exporting symbols (dynamic library)

  • В MSVC по-умолчанию никакие — надо перечислять:

    • при компиляции

    • Module Definition Files

    • __declspec(dllexport)

    • …​

  • В GCC — все

    • → можно скрывать __attribute__((visibility("hidden")))

usage: explicit linking

  • MSVC:

    • LoadLibrary*, GetProcAddress

typedef void(*func_pointer)(int, double);

HMODULE lib = LoadLibrary("some.dll"); // or LoadLibraryEx
auto func = reinterpret_cast<func_pointer>(GetProcAddress(lib, "func_name_in_chars"));

FreeLibrary(lib);

  • Linux: dlopen, dlsym, dlclose

  • есть хуки на загрузку/выгрузку библиотек (*)

implicit linking

  • компиляция → .lib + .dll. Статическая часть:

    • секции, задающие dynamic-зависимости

    • код оберток, проксирующих вызовы (зачастую оптимизируется)

notes

  • если нет уверенности в совпадении ABI:

    • функции стоит помечать extern "C" и фиксировать calling convention

    • перегрузки функций, методы классов, обработка исключений — могут не работать

      • ⇒ стоит избегать фичей плюсового рантайма

  • если и исполняемый файл и динамическая библиотека собираются в одном проекте — одинаковые компилятор/флаги сборки/платформа, — то совпадение ABI есть
